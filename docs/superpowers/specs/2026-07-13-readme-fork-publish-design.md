## 4. 高级渠道调度机制

### 4.1 启用条件与旧逻辑兼容

文档需要解释：

- 高级调度器默认关闭，关闭时完整沿用原有渠道选择与重试逻辑。
- 调度器候选加载失败时，转发链路回退到旧逻辑，并记录系统错误，避免因为新功能不可用而直接中断请求。
- 渠道级开关可控制单个渠道是否参与调度器。

### 4.2 候选渠道与优先级分桶

文档需要解释：

- 候选渠道同时兼容内存缓存和数据库直查路径。
- 候选结果按 `priority` 从高到低分桶。
- 同一优先级内保留原有权重选择语义。
- Advanced Custom 渠道仍按请求路径过滤。
- 当前优先级耗尽后，只有启用“优先级降级”时才进入下一优先级。

### 4.3 同渠道连续重试与失败阈值

文档需要解释：

- 默认行为是在同一渠道上连续重试，直到达到该渠道的失败阈值。
- 全局默认失败阈值为 3，允许范围为 1 到 100。
- 渠道可以继承全局阈值，也可以设置自己的阈值。
- 关闭“同渠道连续重试”后，会在同优先级候选之间更快轮换。
- 调度器具有内部尝试预算和保险丝，避免异常候选集、错误配置或并发状态导致无界循环。

### 4.4 错误分类与故障处置

文档需要说明调度器并非对所有错误一律禁用渠道，而是根据错误类型选择：

- 停止请求，不继续重试。
- 重试当前渠道。
- 切换渠道但不临时禁用当前渠道。
- 计入失败阈值，达到阈值后进入临时禁用流程。
- 对需要立即故障转移的错误，可跳过常规累计过程，但仍进入受保护的禁用判断。
- 建连或请求发送失败不会被错误地当作可安全禁用渠道的上游业务失败。

状态码规则需要写明：

- 自动重试和自动禁用状态码支持单值与范围配置，如 `401,429,500-503`。
- 输入会进行排序、相邻区间合并和 100 到 599 范围校验。
- `504`、`524` 和特定坏响应体错误具有强制跳过重试保护。
- 默认规则保持与原有重试行为兼容。

### 4.5 重试随机抖动

文档需要说明：

- 可以在每次调度器重试前增加随机延迟，减少集中故障时的瞬时重试风暴。
- 最小值和最大值同时为 0 时关闭抖动。
- 有效非零范围为 100 到 10000 毫秒，并要求最小值不大于最大值。
- 非法区间会回退为不启用抖动，不让错误配置破坏主请求链路。

### 4.6 临时禁用

文档需要解释状态变更的完整机制：

- 默认临时禁用时长为 7200 秒，即 2 小时。
- 渠道可以继承全局禁用时长，也可以单独覆盖。
- 达到阈值后，渠道状态、`auto_disabled_until`、状态原因和对应 Ability 可用状态在同一数据库事务内更新。
- 更新使用带前置状态的 CAS 语义，只允许从启用状态进入调度器临时禁用状态。
- 并发的手动禁用、旧式自动禁用或另一请求发起的新一轮禁用不会被覆盖。
- 数据库更新成功后同步更新内存缓存，保持缓存和数据库状态一致。
- 如果全局设置要求尊重渠道 `auto_ban` 开关，则关闭自动封禁的渠道只会从当前请求排除，不会被调度器临时禁用。

### 4.7 自动恢复与手动恢复

文档需要解释：

- 系统任务调度器定期检查已到期的调度器临时禁用渠道。
- 恢复处理器以 1 分钟间隔参与任务调度，系统空闲时不会持续创建无意义任务记录。
- 关闭高级调度器后，已经存在的临时禁用渠道仍可按到期规则恢复。
- 自动恢复只处理 `status=auto_disabled`、`auto_disabled_until>0` 且已到期的渠道。
- 手动禁用渠道和旧式无到期时间的自动禁用渠道不受该任务影响。
- 渠道可以单独关闭到期自动恢复。
- 恢复同样使用事务和 CAS 条件，并校验扫描时的到期时间，防止旧恢复任务覆盖新一轮禁用。
- 手动恢复受渠道级“允许手动恢复”开关约束，并产生管理员审计记录。

## 6. 调度日志、统计与维护

### 6.1 日志事件

明确列出五类事件：

- `failure`：单次渠道失败。
- `observe_disable`：达到禁用条件但因观察或保护规则未实际禁用。
- `auto_disable`：达到条件并完成临时禁用。
- `auto_recover`：到期自动恢复。
- `manual_restore`：管理员手动恢复。

### 6.2 日志字段和查询

说明日志位于主数据库的 `channel_scheduler_logs` 表，并可记录或筛选：

- 请求 ID、用户、Token、分组、模型。
- 渠道 ID、名称、类型、优先级。
- 尝试次数、禁用时长、禁用截止时间。
- HTTP 状态码、错误代码、错误类型和脱敏原因。
- 已使用渠道和扩展元数据。
- 时间范围与分页。

说明统计接口提供事件总数、各事件数量，以及按失败次数排序的渠道统计。

### 6.3 日志可靠性和清理

说明：

- 调度日志写入失败只进入系统错误日志，不阻断转发主链路。
- 管理员可以按时间删除旧调度日志。
- 系统任务使用分批删除方式处理旧日志，并支持取消和租约心跳，降低长任务对数据库的冲击。
- 日志配置、渠道配置、手动恢复和日志清理均写入管理审计日志。

## 8. 相关转发韧性改进

README 单独说明这些改动与调度器的关系，但不将其误写成调度器核心功能。

### 8.1 空响应识别

说明对 OpenAI Chat、OpenAI Responses、Gemini 和 Claude 等响应格式增加可交付内容检查。真正没有文本、工具调用、媒体或其他有效载荷的响应会转换为统一空响应错误，使重试和故障转移逻辑能够正确处理，而不是向客户端返回表面成功但无内容的结果。

### 8.2 Header 继承与安全透传

说明：

- 渠道 Header 覆盖支持 API Key 和客户端 Header 占位符。
- 支持显式 Header 名、全部 Header 和正则规则透传。
- Hop-by-hop、Host、Cookie、认证、长度、WebSocket 握手等敏感或协议相关 Header 不会被不安全地盲目转发。
- 正则表达式会缓存，空规则和非法表达式会返回明确配置错误。
- 适配器 Header、透传 Header 和显式覆盖具有确定的合并顺序。

### 8.3 请求体与流式响应边界

说明本分支补强了请求体复用、Content-Length 处理、流式响应 Header 与生命周期保持，减少重试时请求体已消费、上游 Header 丢失或流提前结束的问题。
